3 Architectural Approaches

Each has different trade-offs for complexity, SEO, and scalability

A

Full Static (JAMstack)

Next.js generates all pages at build time using Google Books API. Deployed on Vercel for free.

Pros

  • Blazing fast, perfect SEO
  • Free hosting (Vercel)
  • Zero backend to maintain

Cons

  • Book data can go stale
  • Slow builds as site grows
  • Search must call API client-side
B ⭐ Recommended

Next.js with ISR + API Routes

Book/category pages use Incremental Static Regeneration — pre-built but auto-refreshed. API Routes proxy Google Books so keys stay server-side.

Pros

  • Always fresh data, great SEO
  • Search is fast via API routes
  • API key never exposed to browser
  • Still deployable on Vercel free tier

Cons

  • Slightly more setup than pure static
  • Cold starts on API routes (minimal)
C

Next.js + Separate Backend + Database

Next.js frontend + FastAPI/Express backend + PostgreSQL to cache books, track searches, and aggregate multiple APIs.

Pros

  • Maximum flexibility
  • Can mix multiple book data sources
  • Analytics & personalisation later

Cons

  • Much more infrastructure to maintain
  • Higher cost (database server)
  • Overkill for an MVP
